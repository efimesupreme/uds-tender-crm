# Дорожная карта MVP

## Этап 0. Подготовка

Результат:

- документация в репозитории;
- задачи в Issues;
- выбран стек;
- зафиксированы ограничения MVP.

## Этап 1. Интерфейс на моковых данных

Результат:

- приложение запускается локально;
- есть навигация;
- есть дашборд;
- есть реестр заявок;
- есть карточка заявки;
- есть страница «Мои задачи»;
- есть справочники;
- используются тестовые данные.

Цель этапа — проверить логику и удобство до подключения базы.

Дополнительный санитарный шаг перед визуальной системой: выполнены мелкие исправления доступности и форм перед визуальной системой — подписи полей, обязательность, понятные ошибки, alert-сообщения и доступные назначения кнопок без изменения бизнес-логики и визуального стиля.

## Этап 2. Типы и структура данных

Результат:

- строгие TypeScript-типы;
- единая модель заявки;
- единая модель задач;
- моковые данные соответствуют будущей базе.

## Этап 3. База данных

Результат:

- подключён Supabase;
- создана схема таблиц;
- добавлены seed-данные;
- интерфейс читает данные из базы.

## Этап 4. CRUD заявок

Результат:

- создание заявки;
- редактирование заявки;
- смена статуса;
- фиксация причины отказа;
- автоматическая запись истории статусов.

## Этап 5. Задачи и сроки

Результат:

- создание задач по заявке;
- сроки задач;
- просрочки;
- фактическое закрытие;
- возврат на доработку;
- страница «Мои задачи» работает как рабочий список.

## Этап 6. Журнал событий и аудит

Результат:

- ключевые действия пишутся в журнал;
- история статусов видна в карточке;
- можно восстановить ход обработки заявки.

## Этап 7. Дашборд

Результат:

- задачи на сегодня;
- просрочки;
- заявки без следующего действия;
- КП на согласовании у МЛ;
- ближайшие сроки подачи;
- базовая аналитика по отказам и результатам.

## Этап 8. Импорт текущего монитора

Результат:

- загрузка CSV/XLSX или подготовленного CSV;
- предпросмотр;
- сопоставление колонок;
- создание карточек заявок;
- фиксация события импорта.

## Этап 9. Тестирование на двух пользователях

Результат:

- Тагиев создаёт и ведёт заявку;
- Безрукова видит и закрывает свои задачи;
- одна тестовая заявка проходит полный путь;
- система заменяет таблицу в тестовом режиме.

## Этап 10. Внедрение

Предлагаемый сценарий:

1. Перенести текущие заявки.
2. Одну неделю вести таблицу и систему параллельно.
3. Сравнить расхождения.
4. Исправить критичные неудобства.
5. Сделать систему основным инструментом.
6. Таблицу оставить архивом.

## Этап 11. Канбан заявок и суммы по стадиям

Результат:

- на странице заявок доступно переключение между таблицей и канбаном;
- заявки сгруппированы по укрупнённым стадиям процесса УДС-Проект;
- в заголовке каждой стадии видно количество заявок и сумму КП;
- карточки можно переносить между целевыми стадиями через правила процесса;
- недопустимые переходы блокируются без обхода бизнес-логики.

## Этап 12. Автоматические сценарии процесса

Результат:

- предложения следующего действия рассчитываются на лету по карточкам заявок;
- на дашборде есть блок «Предлагаемые действия» с фильтрацией под Тагиева и Безрукову;
- в карточке заявки есть блок «Предлагаемые действия по заявке»;
- из предложений можно создавать задачи на подготовку КП, согласование КП с МЛ, подачу КП и обратную связь;
- store защищает автоматические действия от дублей активных задач;
- выбор текущего demo-пользователя перенесён из левого меню в правый верхний угол рабочего интерфейса.

## Этап 13. Технический аудит и очистка перед визуальной системой и постоянным хранилищем

Результат:

- временный слой `localStorage` явно отделён от постоянной бизнес-логики;
- правила процесса, переходов, обязательных проверок, автоматических предложений, рабочих папок и метрик остаются в профильных модулях `lib`;
- store содержит технические helper-функции для событий, истории статусов, генерации идентификаторов и иммутабельного обновления сущностей;
- GitHub Pages сохраняет текущую демо-схему static export и клиентское открытие runtime-created заявок;
- добавлена явная команда `npm run typecheck` перед следующими визуальными и архитектурными PR.

## Этап 12. Polishing layout и дашборда

Результат:

- добавлена фиксированная верхняя полоса с названием раздела, слотом действий и переключателем пользователя;
- левое меню уплотнено и сделано визуально непрерывным по высоте;
- интерфейс получил более компактную рабочую плотность;
- дашборд очищен от лишних вспомогательных блоков и собран в структуру KPI → план/факт + воронка → структура портфеля.
