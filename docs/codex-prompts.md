# Промты для Codex / Cursor

Разработку лучше вести короткими шагами. Не просить Codex сделать всю CRM сразу.

## Промт 1. Скелет интерфейса на моковых данных

```text
Создай MVP веб-приложения для внутренней системы контроля входящих заявок и тендеров.

Технологии: Next.js, TypeScript. Пока без реальной базы данных, используй моковые данные в коде.

Нужны страницы:
- Дашборд;
- Заявки;
- Карточка заявки;
- Мои задачи;
- Справочники.

Сущности на моках:
- заявки;
- задачи;
- история статусов;
- события;
- ссылки на файлы;
- пользователи;
- внешние участники.

Сделай простую навигацию, аккуратный табличный реестр заявок и карточку заявки с блоками:
- шапка;
- базовые данные;
- решение об участии;
- обращение и папка;
- затраты;
- договор и протокол;
- документы;
- КП и подача;
- результат;
- задачи;
- журнал событий.

Не подключай Supabase, авторизацию и сложные права на этом шаге. Главное — рабочий интерфейс на моковых данных.
```


## Промт 3. Реестр заявок

```text
Улучши страницу реестра заявок.

Нужно:
- таблица заявок;
- фильтры по статусу, решению об участии, ответственному, заказчику, региону, виду работ и результату;
- быстрый поиск по названию и заказчику;
- подсветка просроченных заявок;
- отображение следующего действия;
- переход в карточку заявки.

Данные пока остаются моковыми.
```

## Промт 4. Карточка заявки

```text
Доработай карточку заявки.

Нужно показать все блоки из docs/screens.md. Особое внимание:
- общий статус заявки;
- параллельные задачи по затратам, договору, документам, КП;
- ссылка на рабочую папку;
- история статусов;
- журнал событий;
- следующее действие;
- просрочки задач.

Данные пока моковые. Не добавляй реальную базу.
```

## Промт 5. Дашборд

```text
Создай дашборд по моковым данным.

Покажи:
- задачи на сегодня;
- просроченные задачи;
- заявки без следующего действия;
- ближайшие сроки подачи;

Не делай сложные графики. Достаточно карточек, списков и простых таблиц.
```

```text
Подключи Supabase как базу данных и авторизацию.

Не добавляй сложную ролевую модель. Достаточно admin/user.
```

## Промт 10. Полировка MVP

```text
Проведи ревизию MVP.

Проверь:
- активная заявка без следующего действия выводится как проблемная;
- просроченные задачи подсвечиваются;
- отказные заявки сохраняются и фильтруются;
- история статусов пишется автоматически;
- журнал событий понятен;
- карточка заявки не перегружена;
- интерфейс можно использовать вдвоём: Денис и Катя.

Не добавляй новые большие функции, исправляй только критичные пробелы MVP.
```
