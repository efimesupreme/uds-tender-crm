# Процесс и статусы

## 1. Главный принцип

У заявки есть общий статус, но после согласования участия процесс идёт параллельно. Поэтому общий статус не должен скрывать состояние отдельных задач.

Пример: заявка в общем статусе «Подготовка материалов», но внутри:

- затраты — в работе;
- договор — анализ получен;
- документы — просрочены;
- рабочая папка — создана.

## 2. Основные статусы заявки

1. `new` — Новая заявка.
2. `participation_decision` — На решении об участии.
3. `not_participating` — Не участвуем.
4. `participation_approved` — Участие согласовано.
5. `appeal_and_folder` — Заведение обращения и папки.
6. `materials_preparation` — Подготовка материалов.
7. `materials_received` — Материалы получены.
8. `internal_approval` — Внутреннее согласование.
9. `costs_approved` — Затраты утверждены.
10. `offer_preparation` — КП в подготовке.
11. `owner_approval` — КП на согласовании у МЛ.
12. `ready_to_submit` — КП готово к подаче.
13. `submitted` — КП подано / предложение направлено.
14. `feedback_waiting` — Ожидание обратной связи.
15. `won` — Победили.
16. `lost` — Проиграли.
17. `withdrawn_after_start` — Отказались после запуска подготовки.
18. `missed_deadline` — Не успели податься.
19. `canceled_or_paused` — Тендер отменён / заморожен.

## 3. Правила переходов

### Новая заявка → На решении об участии

Действие выполняет Тагиев. Система фиксирует время направления на решение ГД.

### На решении об участии → Не участвуем

Обязательные поля:

- причина неучастия;
- комментарий.

### На решении об участии → Участие согласовано

Система фиксирует дату решения. После этого нужно создать задачи:

- завести обращение;
- создать рабочую папку;
- подготовить затраты;
- проанализировать договор;
- собрать комплект документов.

### Участие согласовано → Заведение обращения и папки

После согласования участия обязательно заводится обращение и создаётся рабочая папка заявки. Этап нужен, чтобы не терять внутренний номер и ссылку на рабочую папку.

Минимальные поля:

- номер обращения;
- ссылка на корневую рабочую папку;
- дата создания папки;
- ответственный;
- статус задачи.

В заявке хранится только корневая ссылка на папку обращения. Типовые подпапки для исходных данных, тендерной документации, проекта договора, затрат, протокола разногласий, КП, подачи и обратной связи рассчитываются автоматически по шаблону, поэтому отдельные постоянные ссылки на эти подпапки не являются обязательными полями заявки.

### Заведение обращения и папки → Подготовка материалов

После запуска рабочих задач заявка находится в подготовке материалов.

### Подготовка материалов → Материалы получены

Переход возможен, когда получены ключевые результаты:

- затраты;
- анализ договора или отметка, что проект договора отсутствует;
- комплект документов или перечень недостающих документов.

### Материалы получены → Внутреннее согласование

На этом этапе Тагиев проверяет затраты, готовит протокол разногласий и согласовывает его.

### Внутреннее согласование → Затраты утверждены

После совещания с ГД, РП/ГИП и Безруковойй затраты считаются утверждёнными.

### Затраты утверждены → КП в подготовке

Тагиев передаёт утверждённые затраты Безруковой.

### КП в подготовке → КП на согласовании у МЛ

Безрукова подготовила КП и направила его на согласование собственнику.

### КП на согласовании у МЛ → КП готово к подаче

КП согласовано. Если КП вернулось на доработку, увеличивается счётчик возвратов.

### КП готово к подаче → КП подано / предложение направлено

Безрукова подала КП на площадку или передала его ответственному за контакт с заказчиком.

### КП подано → Ожидание обратной связи

Система фиксирует дату следующего касания.

### Ожидание обратной связи → Финальный статус

Возможные финальные статусы:

- победили;
- проиграли;
- тендер отменён / заморожен;
- нет обратной связи;
- отказались после запуска подготовки.

## 4. Статусы задач

У задач используется единая модель без параллельных технических статусов:

- `new` — «Новая»;
- `in_progress` — «В работе»;
- `completed` — «Выполнено».

Ответственный хранится в `assigneeUserId`. Для задач управленческого контура по умолчанию назначается Тагиев (`u-denis`, Тагиев), для задач КП/документов/подачи — Безрукова (`u-katya`, Безрукова). Ответственного можно вручную поменять в списке «Мои задачи»; изменение сохраняется в клиентском store и фиксируется событием журнала.

Исполнение задачи выполняется кнопкой «Исполнено» после подтверждения. Система записывает `completedAt`, рассчитывает фактическую длительность от `startedAt` или `createdAt`, пишет событие `task_completed` и применяет связанные эффекты процесса: создаёт следующий допустимый шаг без дублей, обновляет рабочие признаки заявки или показывает предупреждение, если для движения не хватает данных в карточке.

## 5. Обязательное поле следующего действия

Для активной заявки должно быть заполнено:

- что делаем дальше;
- кто ответственный;
- до какой даты.

Активная заявка без следующего действия считается проблемной и должна выводиться на дашборд.

## 4. Правила закрытия заявки

Закрытие заявки выполняется только через блок **«Результат и закрытие»** в карточке заявки. Прямой перевод в финальные статусы через канбан или быстрые операционные переходы не используется, потому что закрытие должно сохранять содержательные данные результата.

Финальные статусы централизованы в `lib/workflow.ts`:

- `not_participating` — не участвуем;
- `won` — победили;
- `lost` — проиграли;
- `withdrawn_after_start` — отказались после запуска подготовки;
- `missed_deadline` — не успели податься;
- `canceled_or_paused` — тендер отменён / заморожен.

Обязательные поля:

- для `not_participating` — причина и комментарий;
- для `lost` — причина проигрыша, комментарий и наша цена, если сумма КП уже известна в заявке;
- для `won` — дата результата и комментарий;
- для `withdrawn_after_start` — причина и комментарий;
- для `missed_deadline` — причина и комментарий;
- для `canceled_or_paused` — причина и комментарий.

При закрытии система меняет статус, заполняет поля результата/закрытия, добавляет запись в историю статусов и пишет событие `request_closed`. Дополнительно фиксируются события результата: `request_won`, `request_lost`, `request_withdrawn` или `not_participating_recorded`.

## Ведение рабочих блоков в карточке

Общий статус заявки остаётся верхнеуровневым этапом процесса, но ежедневная работа фиксируется в отдельных блоках карточки заявки: решение об участии, затраты, договор/протокол, документы, КП/подача и обратная связь. Сохранение блока не переводит заявку автоматически в другой статус, а добавляет событие в журнал. Исключение — синхронизация ключевых рабочих значений с уже существующими полями заявки: `costAmount`, `offerAmount`, `plannedMarginPercent`, `nextActionDueAt`, `nextActionText`.

## Демо-роли без настоящей авторизации

В текущем статическом демо Тагиев управляет процессом: контролирует решение об участии, обращение и рабочую папку, просрочки, узкие места и следующие действия. Безрукова ведёт коммерческий контур: КП, документы, согласование с МЛ, подачу и обратную связь после подачи.

Настоящая авторизация, Supabase, API routes и серверные действия не подключаются на этом этапе. Выбранный в интерфейсе demo-пользователь сохраняется на клиенте и используется только как actor для действий и фильтров рабочих зон. Полноценные права доступа будут добавлены позже.

## Автоматические сценарии процесса

Автоматические сценарии не меняют статус заявки без подтверждения пользователя. Модуль `lib/process-automation.ts` рассчитывает предложения на лету по текущей заявке, активным задачам и датам.

Правила MVP:

- участие согласовано, но нет номера обращения или рабочей папки — предложить Тагиеву завести обращение и папку;
- затраты утверждены, но подготовка КП ещё не начата — предложить создать задачу Безруковой на подготовку КП;
- КП готово, но не передано на согласование МЛ — предложить задачу на согласование КП с МЛ;
- КП согласовано МЛ, но не подано — предложить задачу Безруковой на подачу КП;
- КП подано, но нет следующего касания — предложить задачу на обратную связь;
- дата следующего касания прошла — предложить запросить обратную связь;
- документы не готовы, а срок подачи ближе 3 дней — подсветить ускорение комплекта документов.

Перед созданием задачи store проверяет наличие активной задачи такого же типа по этой заявке. Если задача уже есть в статусе `new`, `in_progress`, `waiting` или `returned`, дубль не создаётся, а предложение исчезает после пересчёта.

## Обязательные данные перед переходами

Перед изменением статуса система сначала проверяет допустимость перехода по `workflow.ts`, а затем проверяет готовность данных через `lib/transition-guards.ts`. Ошибки блокируют переход, предупреждения не блокируют, но показываются пользователю.

Ключевые блокировки:

- переход к решению об участии требует название, заказчика, вид работ, источник и ответственного;
- согласование участия требует решения «участвуем» и комментарий или дату решения ГД;
- переход после блока обращения требует номер обращения и ссылку на корневую рабочую папку;
- подготовка материалов требует обращения, папки и задач по затратам, договору и документам;
- утверждение затрат требует статус затрат «утверждено» и сумму затрат или комментарий, почему сумма не нужна;
- подготовка и согласование КП требуют утверждённые затраты, готовность КП, сумму КП и подтверждающую дату или комментарий;
- готовность к подаче и подача требуют согласованное КП, готовые документы, сумму КП, дату подачи и исполнителя подачи;
- ожидание обратной связи требует дату подачи и следующий шаг или дату следующего касания.

Финальные статусы по-прежнему выставляются только через закрытие заявки с причиной и комментарием.

## Исполнение задач и процессные эффекты

Во вкладке «Мои задачи» задачи имеют единые статусы `new`, `in_progress`, `completed`, которые отображаются как «Новая», «В работе», «Выполнено». Новая задача сначала переводится кнопкой «Взять в работу», а для задачи в работе кнопка «Исполнено» открывает подтверждение; после подтверждения store записывает `completedAt`, фактическую длительность и событие журнала.

Функция `completeTaskWithEffects` применяет связанные действия без дублей: утверждённые затраты могут создать подготовку КП, готовое КП — согласование с МЛ, согласованное КП — подачу. Договорная цепочка укрупнена до подготовки протокола разногласий без отдельных задач согласования с юристами и ГД. Если в карточке не хватает обязательных данных (решение об участии, обращение и папка, готовность документов, готовность или согласование КП, дата подачи), задача всё равно завершается, но пользователь видит предупреждение о данных, нужных для движения процесса. Переход статуса заявки выполняется только если он разрешён процессом и `transition-guards`.

## Укрупнённая логика задач «Мои задачи»

Актуальный справочник типов задач: «Согласовать участие с ГД», «Завести обращение», «Утвердить затраты», «Проанализировать договор», «Подготовить протокол разногласий», «Собрать комплект документов», «Подготовить КП», «Согласовать КП с МЛ», «Подать КП», «Прочее». Старые технические шаги создания папки, проверки затрат, отдельных согласований протокола, управленческого решения, уточнения документов и фиксации результата больше не используются как отдельные типы задач.

Статусы задач единые: `new` («Новая»), `in_progress` («В работе»), `completed` («Выполнено»). Для новой задачи основное действие — «Взять в работу»: задача переходит в работу, фиксируется `startedAt` и событие журнала. Для задачи в работе действие — «Исполнено»: открывается простое подтверждение, после подтверждения фиксируется `completedAt`, событие журнала и запускаются процессные эффекты. Для выполненной задачи доступно только второстепенное действие «Вернуть в работу»; дата выполнения сохраняется в модели как история предыдущего завершения, а статус возвращается в работу.

Процессные эффекты после выполнения: «Завести обращение» проверяет номер обращения и ссылку на папку; «Утвердить затраты» при утверждённых затратах создаёт «Подготовить КП» без дублей; «Проанализировать договор» создаёт «Подготовить протокол разногласий»; «Подготовить протокол разногласий» пишет событие готовности договорной части; «Собрать комплект документов», «Подготовить КП», «Согласовать КП с МЛ» и «Подать КП» проверяют готовность соответствующих полей карточки и либо создают следующий укрупнённый шаг, либо показывают предупреждение. «Прочее» не запускает процессных эффектов.

## Демо-учётные записи и видимость задач

В demo-store доступны три учётные записи: Тагиев (`u-denis`) — руководитель процесса, Безрукова (`u-katya`) — КП, документы и подача, Админ (`u-admin`) — администратор системы. Исторические id `u-denis` и `u-katya` сохранены для совместимости с данными в `localStorage`.

Во вкладке «Мои задачи» Тагиев видит только задачи с `responsibleUserId`/`assigneeUserId = u-denis`, Безрукова — только задачи с `responsibleUserId`/`assigneeUserId = u-katya`, а Админ видит весь список задач. Срезы «Новые», «В работе», «Выполнено» применяются после этого правила видимости.
