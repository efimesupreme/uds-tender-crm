# Импорт текущего монитора

## 1. Цель

Перенести текущий монитор тендеров из Excel/Google Sheets в карточки заявок без ручного переписывания.

## 2. Текущие колонки монитора

В текущем файле используются ключевые поля:

- ID;
- Наименование;
- Заказчик;
- Регион;
- Ссылка b2b;
- Срок подачи;
- Статус подачи;
- Решение об участии;
- Затраты;
- Протокол разногласий;
- Результат.

## 3. Сопоставление с новой системой

| Текущая колонка | Поле в системе | Комментарий |
|---|---|---|
| ID | internal_number | Если ID отсутствует или равен «-», создать временный номер |
| Наименование | title | Переносить полностью, переносы строк сохранить |
| Заказчик | customer_name | В будущем можно нормализовать в справочник клиентов |
| Регион | region | Свободный текст на MVP |
| Ссылка b2b | source_url | Сохранять как ссылку на источник |
| Срок подачи | submission_deadline_at / deadline_comment | В файле встречаются даты и текстовые комментарии, нужна ручная проверка |
| Статус подачи | current_status / submission_status_comment | Использовать для первичного определения статуса |
| Решение об участии | participation_decision | Участвуем / Не участвуем / На рассмотрении |
| Затраты | costs_status_comment | На MVP перенести как комментарий к блоку затрат |
| Протокол разногласий | protocol_status_comment | На MVP перенести как комментарий к блоку договора |
| Результат | result_comment / result_status | Использовать для определения результата и причины |

## 4. Правила определения статуса

Примерные правила:

- если решение об участии = «Не участвуем», статус = `not_participating`;
- если статус подачи содержит «КП подано» или «подано», статус = `submitted` или `feedback_waiting`;
- если результат содержит «проиграли», статус = `lost`;
- если результат содержит «отказ», «не стали подаваться», статус = `withdrawn_after_start` или `not_participating`;
- если результат содержит «не успели», статус = `missed_deadline`;
- если решение об участии содержит «на рассмотрении», статус = `participation_decision`.

Все автоматические правила должны быть проверяемыми вручную.

## 5. Причины неучастия

Из свободного текста результата нужно попытаться выделить типовую причину:

- не наш предмет;
- нет ресурсов;
- короткий срок;
- другой регион;
- невыгодный договор;
- нет проекта договора;
- нет исходных данных;
- неконкурентная цена;
- требуется СМР;
- требуется личное присутствие;
- нет контакта с заказчиком;
- стратегически неинтересно;
- прочее.

Свободный текст результата сохранять полностью в комментарии.

## 6. Что не переносить автоматически

Не нужно пытаться автоматически восстановить историю статусов по старому файлу. Для импортированных заявок можно создать одно событие: `imported_from_legacy_monitor`.

## 7. Критерии успешного импорта

- все строки с заявками созданы как карточки;
- ссылки сохранены;
- сроки подачи перенесены или вынесены в комментарий;
- решение об участии сохранено;
- отказные заявки не потеряны;
- исходный результат сохранён в комментарии;
- импортированные заявки можно фильтровать в реестре.
