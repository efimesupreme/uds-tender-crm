# Метрики процесса

Демо-CRM теперь считает не только статус заявки, но и длительность этапов, сроки задач и потенциальные узкие места процесса. Расчёты выполняются на клиенте по данным из `localStorage` и моковым данным, без Supabase, API routes и серверных действий.

## Что считается

- **Длительность этапов заявки** — сколько времени заявка находилась в каждом статусе.
- **Длительность текущего этапа** — сколько времени прошло с последней смены статуса до текущего момента.
- **Длительность задачи** — время от создания задачи до завершения или до текущего момента, если задача активна.
- **Просрочка задачи** — насколько фактическое завершение или текущий момент позже планового срока.
- **Средние сроки этапов** — средняя длительность завершённых прохождений по статусам.
- **Узкие места процесса** — список ситуаций, где заявка или задача требуют внимания.

## Как считается длительность этапа

Источник данных — `statusHistory`.

1. История конкретной заявки сортируется по `changedAt` от старых записей к новым.
2. Начало этапа — дата `changedAt` записи перехода в статус.
3. Окончание завершённого этапа — `changedAt` следующей записи в истории этой заявки.
4. Окончание текущего этапа — текущий момент `now`.
5. Длительность — разница между окончанием и началом в миллисекундах.
6. Для интерфейса длительность форматируется в человекочитаемый вид: `35 мин`, `2 ч 10 мин`, `1 день 4 ч`, `5 дней`.

Если история неполная, расчёт показывает только те этапы, для которых есть записи в `statusHistory`. Текущий этап дополнительно может быть рассчитан от `createdAt`, если в истории нет подходящей записи.

## Как считается длительность задачи

Источник данных — задача из `tasks`.

- Начало задачи — `createdAt` задачи.
- Если задача завершена, окончание — `completedAt`.
- Если задача активна, окончание — текущий момент `now`.
- Если у старых демо-записей нет `createdAt`, используется безопасный fallback из доступных дат, чтобы интерфейс не ломался.

Просрочка считается отдельно:

- плановый срок — `plannedDueAt`;
- контрольная дата — `completedAt` для завершённых задач или `now` для активных;
- если контрольная дата позже планового срока, задача считается просроченной, а разница показывается как длительность просрочки.

## Как определяется узкое место

Модуль метрик возвращает потенциальные узкие места по следующим правилам:

- активная заявка без следующего действия;
- активная задача с просроченным плановым сроком;
- активная заявка слишком долго находится в текущем статусе;
- срок подачи близко, а заявка ещё не находится в статусе `submitted`;
- заявка зависла на этапе `owner_approval` — КП на согласовании у МЛ;
- участие согласовано, но не заполнены номер обращения или рабочая папка.

Пороговые значения в демо-версии заданы в коде метрик и нужны для оперативной подсветки проблем, а не для регламентного SLA.

## Где отображаются метрики

- В карточке заявки есть блок **«Сроки этапов»** с историей статусов, датами начала/окончания, длительностью и текущим этапом.
- Таблица задач в карточке заявки расширена до блока **«Сроки задач»**: показываются плановый срок, фактическое завершение, длительность, просрочка и возвраты.
- На дашборде есть блок **«Узкие места процесса»**.
- На дашборде есть блок **«Средние сроки этапов»** по ключевым этапам процесса.

## Ограничения расчёта на демо-данных

- Данные хранятся в `localStorage`, поэтому расчёты зависят от состояния браузера пользователя.
- Нет серверного времени и единого источника истины; текущий момент берётся на клиенте.
- История до внедрения метрик может быть неполной, поэтому часть средних значений может отображаться как «Недостаточно данных».
- Нет рабочих календарей, праздников и часов доступности исполнителей: считаются календарные миллисекунды.
- Нет настоящих SLA по типам заявок; пороги узких мест являются демонстрационными.
- Моковые данные не содержат реальных коммерческих сведений.
